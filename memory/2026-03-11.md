### Architecture Insight — Status-Driven Nodes (6:20 AM)

**Regis proposed a fundamental rethink** of the graph schema:

> "If each element is a node and we can connect them freely, we don't need specifically to connect an email to a task. The email itself is a node with a field status, so a task by itself."

**The model:**
- Every node can have a `status` field → makes it an action item
- Hierarchy is defined by edges (`part_of`, `child_of`) → determines task vs sub-task vs project
- No artificial type separation between "email" and "task" — same entity, different status/tags
- Eliminates current duplication: email node + task node for same thing becomes ONE node

**Example hierarchy:**
```
[job] Crown Terrace  status:open
  └─ [order] OLS Stone Center  status:placed
  └─ [quote] GLM  status:received
  └─ [payment] Tom McKinstry  status:done
```

**TARSON assessment:** Architecturally sound — this is how Tana/Notion/Roam work. Reduces duplication, more flexible. Suggested:
1. Add `status` + `due_date` fields to node schema
2. Remove email→task pairs, keep one enriched node
3. Dashboard "task view" = filter by `status != null AND status != done`
4. Rebuild now (59 nodes is nothing, retrofitting at 5000 nodes is painful)

**Decision:** Not yet acted on — Regis raised it, TARSON responded positively. **Next step: schema redesign.**
